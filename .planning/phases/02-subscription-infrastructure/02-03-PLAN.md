<objective>
Implement the Stripe webhook event handlers for subscription lifecycle events and invoice events, plus subscription email notifications.

Purpose: This is the core data synchronization pipeline. Stripe is the billing authority; D1 is the sync layer. When subscription events happen in Stripe (created, updated, deleted, payment succeeded, payment failed), webhook handlers update D1 state, create audit trail entries, and trigger customer emails. Without this pipeline, subscription state changes in Stripe would not be reflected in the application.

Output: Rewritten webhook route with async signature verification, extracted subscription and invoice handler files, and subscription email functions in the existing email utility.
</objective>

**Step 1: Create `app/api/webhooks/stripe/handlers/subscription-handlers.ts`**

This file handles `customer.subscription.created`, `customer.subscription.updated`, and `customer.subscription.deleted` events.

Import from: `@/lib/models/mach/subscriptions` (model CRUD), `@/lib/utils/email` (sendSubscriptionEmail), `@/lib/types/subscription` (types), `stripe` (Stripe types).

**handleSubscriptionCreated(subscription: Stripe.Subscription, stripeEventId: string):**
1. Extract from subscription object: `id` (stripe_subscription_id), `customer` (stripe_customer_id), `status`, `current_period_start`, `current_period_end`, `items.data[0].price.id` (stripe_price_id)
2. Look up the subscription plan by matching `stripe_price_id` in subscription_plans table. If no plan found, log warning and return (this means the subscription was created outside our system).
3. Look up customer by stripe_customer_id. Use the customer_subscriptions table's stripe_customer_id or query the customers table's external_references JSON. For now, use the stripe_customer_id from the subscription to find or create the D1 record. The `customer_id` for D1 can be derived from the subscription metadata if set, or from a customer lookup.
4. Call `createCustomerSubscription()` with all fields.
5. Call `createSubscriptionEvent()` with type 'created'.
6. Send 'created' email via `sendSubscriptionEmail()`. Extract customer email from subscription metadata or Stripe customer object. Use `getStripeForWorkers()` to fetch customer details if needed: `stripe.customers.retrieve(subscription.customer)`.

**handleSubscriptionUpdated(subscription: Stripe.Subscription, stripeEventId: string):**
1. Look up existing D1 subscription via `getSubscriptionByStripeId(subscription.id)`.
2. If not found, log warning and return.
3. Detect what changed by checking key fields:
   - **Pause detected:** `subscription.pause_collection` is set (not null) AND the D1 record's status is not already 'paused'. Update status to 'paused', set pause_collection JSON. Create 'paused' event. Send 'paused' email.
   - **Resume detected:** `subscription.pause_collection` is null AND the D1 record's status is 'paused'. Update status to 'active', clear pause_collection. Create 'resumed' event. Send 'resumed' email.
   - **Cancel at period end:** `subscription.cancel_at_period_end === true`. Update cancel_at_period_end on D1 record. Create 'updated' event (not 'canceled' -- that happens on deletion).
   - **Status change:** If subscription.status differs from D1 status (e.g., 'past_due'), update D1 status. Create 'updated' event.
   - **Period change:** Always update current_period_start and current_period_end from the subscription object via `updateSubscriptionPeriod()`.
4. Always update `updated_at` timestamp.

IMPORTANT: Per research, `customer.subscription.paused` and `customer.subscription.resumed` events do NOT fire when using `pause_collection`. Detection happens inside this `updated` handler by checking the `pause_collection` field.

**handleSubscriptionDeleted(subscription: Stripe.Subscription, stripeEventId: string):**
1. Look up D1 subscription via `getSubscriptionByStripeId(subscription.id)`.
2. If not found, log warning and return.
3. Update status to 'canceled', set canceled_at to current ISO timestamp.
4. Create 'canceled' event.
5. Send 'canceled' email.

**Step 2: Create `app/api/webhooks/stripe/handlers/invoice-handlers.ts`**

**handleInvoicePaymentSucceeded(invoice: Stripe.Invoice, stripeEventId: string):**
1. Check if `invoice.subscription` exists (skip non-subscription invoices).
2. Look up D1 subscription via `getSubscriptionByStripeId(invoice.subscription as string)`.
3. If not found, log warning and return.
4. If this is NOT the first invoice (check `invoice.billing_reason !== 'subscription_create'`), it's a renewal:
   - Create 'renewed' event with details JSON containing invoice ID and amount.
   - Send 'renewed' email with amount info.
5. Update subscription period dates from the invoice's period if available.

**handleInvoicePaymentFailed(invoice: Stripe.Invoice, stripeEventId: string):**
1. Check if `invoice.subscription` exists.
2. Look up D1 subscription.
3. If not found, log warning and return.
4. Create 'payment_failed' event with details JSON containing failure reason (`invoice.last_finalization_error?.message` or 'Unknown error') and next retry date (`invoice.next_payment_attempt` as ISO string).
5. Update subscription status to 'past_due' if not already.
6. Send 'payment_failed' email with failureReason and nextRetryDate.

**Step 3: Rewrite `app/api/webhooks/stripe/route.ts`**

Replace the existing POST handler with the new architecture:

```typescript
import { NextRequest, NextResponse } from 'next/server';
import Stripe from 'stripe';
import { verifyWebhookSignature, getWebhookSecret } from '@/lib/stripe';
import { isWebhookEventProcessed, recordWebhookEvent, cleanupOldWebhookEvents } from '@/lib/models/mach/subscriptions';
import { handleSubscriptionCreated, handleSubscriptionUpdated, handleSubscriptionDeleted } from './handlers/subscription-handlers';
import { handleInvoicePaymentSucceeded, handleInvoicePaymentFailed } from './handlers/invoice-handlers';
```

POST handler flow:
1. Read body with `req.text()` ONCE (Workers Request bodies are streams, can only be consumed once).
2. Get `stripe-signature` header. Return 400 if missing.
3. Call `verifyWebhookSignature(body, signature, getWebhookSecret())`. Return 400 if verification fails. This is the SUBI-05 security fix.
4. **Dedup check:** Call `isWebhookEventProcessed(event.id)`. If true, return 200 with `{ received: true, duplicate: true }`.
5. **Inline cleanup:** Call `cleanupOldWebhookEvents()` (delete events older than 7 days). Fire-and-forget (don't await, or wrap in try/catch so failure doesn't block processing).
6. **Route to handler** via switch on `event.type`:
   - `customer.subscription.created` -> `handleSubscriptionCreated(event.data.object as Stripe.Subscription, event.id)`
   - `customer.subscription.updated` -> `handleSubscriptionUpdated(event.data.object as Stripe.Subscription, event.id)`
   - `customer.subscription.deleted` -> `handleSubscriptionDeleted(event.data.object as Stripe.Subscription, event.id)`
   - `invoice.payment_succeeded` -> `handleInvoicePaymentSucceeded(event.data.object as Stripe.Invoice, event.id)`
   - `invoice.payment_failed` -> `handleInvoicePaymentFailed(event.data.object as Stripe.Invoice, event.id)`
   - Keep existing `payment_intent.succeeded`, `payment_intent.payment_failed`, `checkout.session.completed` handlers (move the inline functions to the same file or keep them inline -- they are small enough).
   - `default` -> log unhandled event type.
7. After successful processing, call `recordWebhookEvent(event.id, event.type)`.
8. Return 200 `{ received: true }`.
9. On processing error: log, return 500 `{ error: 'Processing failed' }` (triggers Stripe retry). Do NOT record in dedup table so retry will reprocess.

ANTI-PATTERNS TO AVOID:
- Do NOT call model functions via self-referential HTTP calls (the existing handlePaymentSucceeded does this -- leave that as-is for now, but new subscription handlers must call model functions directly).
- Do NOT use synchronous `constructEvent` -- always `verifyWebhookSignature` (async).
- Do NOT return 200 on processing errors -- return 500 for Stripe retry.
- Do NOT read `req.text()` or `req.json()` more than once.

For email sending within handlers: look up customer details. The Stripe subscription object has a `customer` field (string ID). Use `getStripeForWorkers()` to call `stripe.customers.retrieve(customerId)` to get email and name. Cache or pass these through handler parameters if needed for efficiency.

For the `manageUrl` in emails: construct as `https://beauteas.com/account/subscriptions/${d1SubscriptionId}`. This page will be built in Phase 3.
